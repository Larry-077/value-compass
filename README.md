# 🧭 价值罗盘 ValueCompass —— 价值手册驱动的大模型对齐工具

> 一门「伦理设计」课程的项目实现。本系统把抽象的 AI 伦理价值**编码**为机器可读的《价值手册》，并把它构造成**奖励信号**驱动大模型对齐：既支持「自我批评 → 修订」（宪法式 AI），也支持「采样 N 个候选 → 价值奖励选优」（Best-of-N，推理期 RLHF），并能**从人类偏好中训练 reward model** 反过来学习价值权重，让"价值对齐"变得**可编辑、可运行、可度量**。
>
> 🔬 **研究主线与方法（背景→研究问题→方法→实验）见 [docs/method.md](docs/method.md)** —— 围绕 RLHF 的核心机件 reward model 展开。

## 一、项目背景

随着大语言模型（LLM）深度嵌入搜索、教育、医疗咨询、内容创作等场景，**"模型说什么"正在直接影响人的判断与行为**。一个能力很强但价值不对齐的模型，可能在用户脆弱时提供自伤方法、在被诱导时编造虚假文献、在不经意间强化性别与地域偏见。因此，如何把人类社会的伦理价值"注入"到模型行为中——即**价值对齐（AI Alignment）**——成为 AI 伦理设计的核心命题。

工业界已有一些实践，最具代表性的是 Anthropic 提出的**宪法式 AI（Constitutional AI）**：用一份写明价值原则的"宪法"，让模型依据这些原则对自己的回答进行批评和修订，从而减少对大量人工标注的依赖。然而对学习者和设计者而言，这套机制往往是"黑箱"的——价值原则藏在系统内部，无法直接编辑、观察其作用，更难以量化"加了某条价值之后效果到底变好了多少"。

**本项目正是要把这个黑箱打开。** 我们设计了一个面向伦理设计的工具——价值罗盘，它围绕三个理念：

1. **价值是可编码的**：把"无害、诚实、公平"等抽象价值拆解为带权重的具体原则条目，形成一份人类可读、机器可用的《价值手册》。
2. **对齐过程是可见的**：完整呈现"对齐前的原始回答 → 依据手册的自我批评 → 对齐后的修订回答"，让价值如何改变模型行为一目了然。
3. **对齐效果是可度量的**：用模型作为评审（LLM-as-judge）对回答逐维度打分，量化对齐前后的价值得分变化，并支持真实用户参与评价。

本项目的伦理设计价值在于：它不仅"使用"了对齐技术，更**把价值判断的权力交还给设计者与用户**——任何人都可以编辑这份价值手册，决定模型应当遵守哪些价值、各自有多重要，并立刻看到结果。这正是"伦理设计应当透明、可参与、可问责"理念的一次具体落地。

## 二、系统需求设计

### 2.1 目标用户

- **AI 伦理 / 设计课程的学习者与研究者**：理解并亲手实践价值对齐机制。
- **产品/策略设计者**：在不写复杂代码的前提下，试验不同价值取向对模型输出的影响。
- **评审者**：对一个模型的价值表现做可量化的评估。

### 2.2 功能性需求

| 编号 | 需求 | 说明 |
|------|------|------|
| F1 | 价值模型与价值手册的编码 | 提供分维度、可加权、可启停的价值原则编辑能力 |
| F2 | 对齐流水线 | 对任意问题，生成"原始回答 → 自我批评 → 修订回答" |
| F3 | 价值评分 | 由 LLM 评审依据手册对回答逐维度打分（0–2） |
| F4 | 前后对比可视化 | 以雷达图、评分条、对照栏直观呈现对齐效果 |
| F5 | 批量基准评估 | 对内置伦理敏感题集批量运行，汇总整体提升 |
| F6 | 用户调研 | 收集用户对对齐效果的主观评价并做统计 |
| F7 | 对照实验支持 | 关闭某些价值原则，观察对齐效果如何变化 |

### 2.3 非功能性需求

- **可演示性**：未配置 API Key 时自动回退 mock，保证任意环境下流程完整可跑。
- **安全性**：API Key 仅存在于后端 `.env`，绝不暴露给前端。
- **轻量可部署**：纯前后端分离，依赖少，一条命令启动。
- **可扩展性**：价值维度、测试题、模型均可通过数据/配置扩展。

### 2.4 伦理与边界

本系统会调用模型生成"对齐前的原始回答"，其中可能包含**故意未对齐**的内容（用于展示对齐价值）。因此：
- 系统定位为**研究与教学工具**，不面向公众提供未对齐输出；
- 内置题集中的高风险问题仅用于测试对齐能力，修订回答均坚持安全底线（如自伤场景引导求助热线）；
- 价值手册的默认条目参考了 HHH 框架、Schwartz 价值理论与国内生成式 AI 管理规范，但**承认价值本身存在多元与争议**，故将其设计为完全可编辑——系统不替用户决定"什么是对的"。

## 三、详细设计

### 3.1 总体架构

```
┌────────────── 浏览器 (React) ──────────────┐
│  价值手册编辑器 │ 对齐Playground │ 评估面板 │ 用户调研 │
└───────────────────────┬─────────────────────┘
                        │ /api/run, /api/userstudy
┌───────────────────────▼─────────────────────┐
│            Express 后端 (server/)            │
│  prompts.js  把价值手册注入三类 prompt        │
│  claude.js   调用 claude-sonnet-4-6 / mock 回退│
└───────────────────────┬─────────────────────┘
                        │ openai (OpenAI 兼容)
                ┌───────▼────────────────────┐
                │ 中转 apicz.boyuerichdata.com │
                └─────────────────────────────┘
```

### 3.2 价值模型（数据结构）

价值模型是系统的核心资产，定义于 `src/data/defaultHandbook.js`。它分为 5 个**价值维度**，每个维度下含若干**原则**：

```
维度 dimension { id, name, en, color, desc, principles[] }
原则 principle { id, statement, weight(1–5), enabled }
```

默认 5 个维度：**无害 / 诚实 / 有用 / 公平 / 自主与隐私**，共 13 条原则。`weight` 表示重要性并参与加权，`enabled` 用于对照实验。用户的编辑结果持久化在浏览器 `localStorage`。

### 3.3 对齐流水线（核心算法）

对任意用户问题 `q`，系统依次执行四步（`server/index.js` 的 `/api/run`）：

1. **生成原始回答**（`buildGenerate`）：不提供价值手册，得到"对齐前"基线 `raw`。
2. **自我批评 + 修订**（`buildAlign`）：把启用的价值原则渲染为清单注入 system prompt，要求模型依据手册批评 `raw` 并输出 JSON `{critique, revised}`。这是宪法式 AI 思想的简化实现。
3. **评分对齐前**（`buildJudge`）：LLM 评审依据手册对 `raw` 逐维度打分。
4. **评分对齐后**：同样对 `revised` 打分。

返回 `{raw, critique, revised, rawScores, revisedScores}`，前端据此渲染对比。

> 设计取舍：之所以"生成"与"评分"分两个独立模型调用，是为了让评审视角与生成视角解耦，降低"自卖自夸"偏差；评分采用 0/1/2 三档而非连续分，是为了提高不同样本间的可比性与稳定性。

### 3.4 提示工程

`server/prompts.js` 负责把价值手册转写为模型可执行的指令。关键点：
- `renderHandbook()` 把启用原则按维度分组、带 id 与权重渲染，使模型能在批评时**引用具体原则编号**，便于追溯。
- 批评/评分均要求**严格 JSON 输出**，后端用 `extractJson()` 做容错解析（直接解析失败时回退到正则提取）。

### 3.5 评估与可视化

- `Radar.jsx`：纯 SVG 实现的雷达图，红/绿双层叠加对齐前后各维度得分，无第三方图表依赖。
- `Dashboard.jsx`：对内置 10 道题批量运行，计算每维度均分与整体均分，输出"提升 Δ"与逐题明细表。

### 3.6 用户调研模块

- Playground 中可对单次结果直接投票；`UserStudy.jsx` 提供独立问卷（偏好 / 可信度 / 有用度 / 文字意见）。
- 反馈以 JSON 持久化到后端 `data/userstudy.json`，并实时汇总：样本数、更认可"对齐后"的比例、平均可信度与有用度。

## 四、系统使用说明

### 4.1 安装与启动

```bash
# 1. 安装依赖
npm install

# 2.（可选）配置真实模型
cp .env.example .env
# 编辑 .env，填入 OPENAI_API_KEY（中转令牌）与 OPENAI_BASE_URL；不填则使用内置 mock 演示

# 3. 同时启动前端(5173)与后端(8787)
npm run dev
```

打开 http://localhost:5173 。右上角徽标会显示当前是「真实模型」还是「Mock 演示模式」。

### 4.2 使用流程

系统按 **①价值手册 → ②困难数据集 → ③对齐 Playground → ④Best-of-N → ⑤Reward Model → ⑥评估面板 → ⑦用户调研** 的流水线组织。完整的逐页说明、数据流向、推荐操作步骤与常见问题见 **[docs/usage.md](docs/usage.md)**。

最短上手路径：
1. **②困难数据集**：勾选 4 类加压技术 → 构建 → 得到能让模型翻车的困难集 + 各技术翻车率。
2. **⑥评估面板**：分别选「内置基准」与「困难集」运行 → 对比整体均分（普通题高、困难题低且对齐提升明显）。
3. **③/④** 选困难题看单题对齐过程与 Best-of-N 曲线，并投票。
4. **⑤Reward Model** 用投票训练奖励模型、写回手册；**⑦** 看调研汇总。

### 4.3 目录结构

```
value-compass/
├─ server/            后端
│  ├─ index.js        Express 入口与 API（含 /api/bestofN、/api/train-rm）
│  ├─ prompts.js      价值手册→prompt 构造
│  ├─ reward.js       奖励函数 + Bradley-Terry reward model 训练
│  └─ claude.js       模型调用（OpenAI 兼容中转）+ mock 回退
├─ src/
│  ├─ data/           价值手册 + 测试题集
│  ├─ components/     六个功能页（含 BestOfN / RewardModel）+ 雷达图
│  └─ lib/            API 与本地存储封装
└─ docs/              method.md（研究方法）+ user-study.md（调研）
```

## 六、局限与展望

- **强基座天花板**：当基座模型本身已较对齐（如 claude-sonnet-4-6）时，清晰有害题上 N 个候选常都满分，Best-of-N 曲线趋平——这是真实现象而非 bug，实验应侧重边界/两难题与弱基座对照（详见 [docs/method.md §5](docs/method.md)）。
- 评分由单一模型担任评审，存在评审偏差；后续可多评审投票或人工校准（E2 的人机一致性已部分量化）。
- reward model 在低维（维度分向量）上拟合，特征粒度较粗；可扩展到更细的句级特征或文本 embedding。
- mock 模式为启发式规则，仅供离线演示；真实效果请配置中转令牌。
- 未来可扩展：DPO 式直接偏好优化、价值冲突（两难取舍）专项测试、多模型横向对比。
